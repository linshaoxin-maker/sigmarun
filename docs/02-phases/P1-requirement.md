# P1 Requirement — sigmarun

> 2026-07-10 正式化（取代同名缺口文档）。Mode: Full ｜ 输入：P0 制品 + 语料库 00–25 ｜ 下游：P2 BDD、traceability。
> UC 依标准内嵌本文件 §4；NFR §5；UX §6；安全合规 §7；假设 §8。术语见 [glossary.md](glossary.md)。

## 1. 背景与问题

单人或小团队同时开多个 AI 编程窗口（Claude Code / Codex）时，缺少项目级协作层：任务口头分派、完成靠 agent 自述、窗口互踩文件、会话丢失即上下文丢失、跨 run 决策无传承。已有方案（如 Claude Code agent teams）为纯文件协调但无租约与证据门禁，存在"忘标完成、状态滞后"等已知病症（13 附录 B）。

## 2. 目标与非目标

**目标**：让多个 coding agent 在同一 repo 上并行工作时——任务分派可查、完成有证据、冲突被阻止、掉线可恢复、决策可传承；全程用户保持控制权。
**非目标**（README §5 Out）：不替代 coding agent 智能；不做中心调度 AI/daemon；不跨 repo；不做拜占庭防御；MVP 不自动合 main、不含 Cursor adapter。

## 4. 用例规约（UC-001…009）

> 触发方式统一为 slash 命令（canonical 面见 04 §1.1）；步骤用行为语言，不含实现细节。每个 UC 的 BDD 见 `BDD-UC-XXX.feature`。

### UC-001 规划一个 Run
- 主参与者: 项目负责人；次参与者: planning agent、gateway ｜ Story: US-001 ｜ NFR: NFR-009 ｜ 优先级: P0
- 前置: 项目已 init；工具已装 adapter。
- 主流程: ① 用户给出目标（可选模式 feature/debug/review）② planning agent 读项目与 L4 项目记忆，拆出带目标/验收/路径范围/依赖的任务集 ③ gateway 校验 payload、分配 RUN/TASK ID、落为 draft ④ 向用户呈现任务表与警告、给出下一步。
- 扩展: 3a payload 违规→逐条报错，agent 修正重试一次后仍败则呈报（09 §8）；3b 与既有 run 计划指纹相同→拒绝并指向已有 RUN（D17）；3c 目标文本命中 secret 模式→警告用户改写（warn-only）。
- 后置: 成功=RUN(draft) 与任务索引存在且事件在案；失败=无任何半写状态。
- 业务规则: 引 09 §8 校验表；payload 禁写运行态字段（09 §9）。

### UC-002 发布任务队列
- Story: US-001 ｜ NFR: — ｜ P0
- 前置: RUN 处于 planned 且有 draft 任务。
- 主流程: ① 用户审阅任务表 ② 确认发布（可选子集）③ 任务转 ready、run 激活 ④ 返回可领数量。
- 扩展: 2a 用户要求修改→回 UC-001 调整后再发；3a 与其他 active run 路径重叠→按策略警告或阻断（D18）。
- 后置: ready 任务可被领取。

### UC-003 领取并执行任务
- Story: US-002 ｜ NFR: NFR-001/002/003 ｜ P0
- 前置: run active 且有 ready 任务。
- 主流程: ① 用户在某窗口发起领取（可起名）② 系统注册身份（同名幂等）③ 原子领取最优任务并给出租约与工作区建议 ④ 注入上游上下文必读集 ⑤ agent 在隔离工作区实现，期间可提问/报阻塞/心跳 ⑥ 完成一个任务即停并汇报（连续模式须显式开启）。
- 扩展: 3a 无可领任务→结构化原因（队列空/依赖/冲突/上限）呈报即止；3b 同窗口已持任务→提示先交或释放（上限 1）；5a 沙箱拒绝建工作区→诚实停报并给出人工路径（F-c）；5b 执行中发现任务拆错→发 replan 提议（M33 路径）。
- 后置: 任务 working、租约在案；失败分支不留半写。
- 业务规则: BR-001 可领取判定（P2 §4）。

### UC-004 定向派活
- Story: US-003 ｜ NFR: — ｜ P1
- 前置: 同 UC-003；用户知道目标 TASK。
- 主流程: ① 用户点名某窗口领某任务 ② 系统按 UC-003 流程但跳过挑选、直取该任务。
- 扩展: 2a 该任务被占/依赖未满足/路径冲突→报确切原因并停止，**不得改领他任务**。
- 后置: 同 UC-003。

### UC-005 提交证据
- Story: US-004 ｜ NFR: NFR-004（R-013）｜ P0
- 前置: 任务 working，实现与检查已跑。
- 主流程: ① agent 汇总改动、检查结果原始输出、验收逐条对照、已读上下文确认、交接说明 ② 系统机械校验（覆盖率/一致性/越界/脱敏）③ 通过则任务转 submitted（policy 允许时直通 approved 并留痕）。
- 扩展: 2a 校验失败→逐条缺失呈报，任务留 working；2b 改动越出声明范围→按策略警告或拒绝；2c 输出命中 secret→替换后落盘并计数。
- 后置: 证据可查、下游 handoff 就位。
- 业务规则: "完成 = 通过门禁的 submit"，无副渠道（F1 防线）。

### UC-006 评审与验证
- Story: US-005 ｜ NFR: — ｜ P0
- 前置: 存在 submitted（评审）或 approved（验证）任务。
- 主流程: ① reviewer 角色窗口自主领到评审工作（或人工指定）② 按 checklist 审查证据与改动 ③ 结论=通过/退回（退回须 ≥1 条必改）④ verifier 跑检查并提交结构化结果 ⑤ 通过则任务 verified。
- 扩展: 1a 领审者曾是该任务 owner→拒绝（INV-008，不受任何开关影响）；3a 退回→owner 复工，租约与路径占用无缝（15 §4.2）；4a 验证失败→映射回任务转返工；2a 评审租约过期→回退 submitted 可再领。
- 后置: 每个 approved 任务必有评审记录（含 skip 留痕）。

### UC-007 监工与恢复
- Story: US-006/007 ｜ NFR: NFR-002/005/008/009 ｜ P0
- 前置: run 存在。
- 主流程: ① 用户查看状态：进度、每窗口在干什么、风险、Needs user 清单（每项带可复制命令）② 可选挂常驻巡检器 ③ 需要时执行恢复动作（回收/解阻/修复/暂停）。
- 扩展: 2a agent 静默超时→标 stale，超 3×TTL 自动回收带进展快照，新领取者续做或重做；2b blocked 任务豁免超时判定；3a 怀疑账本被改→审计检出并给修复路径；3b 崩溃残留→repair 按账本前滚/回滚。
- 后置: 用户对"现在该我做什么"零猜测。

### UC-008 集成与留档
- Story: US-008 ｜ NFR: NFR-004 ｜ P0
- 前置: 存在 verified 任务。
- 主流程: ① 用户发起集成 ② 系统按依赖序合并到集成分支，单任务失败即回退该项继续其余 ③ 全量验证后产出报告 ④ 用户可导出脱敏留档 ⑤ 合入主干由用户 PR 完成。
- 扩展: 2a 合并冲突→交集成者处理并记录；4a 导出内容命中 secret→中止并列位置。
- 后置: 集成分支+报告在案；`.team` 外留档可入库。

### UC-009 跨 run 决策传承
- Story: US-009 ｜ NFR: — ｜ P1（读路径随 UC-001/003 在 MVP 生效）
- 前置: 项目记忆文件存在或将建立。
- 主流程: ① run 收尾时系统机械列出决策候选 ② 用户挑选确认 ③ 系统校验出处、盖戳落盘（进 git）④ 之后所有 plan/hydrate 自动携带。
- 扩展: 3a 条目无出处→拒绝；3b 文件超限→警告合并淘汰；3c 过时→新条替换旧条留痕。
- 后置: 新 run 无需人工转述历史决策。

## 7. 安全与合规需求

- **数据分类**：`.team/` 运行态=内部（不入库，D4）；checks 原始输出=按机密处理（落盘前脱敏，R-013）；L4 项目记忆=repo 可见（晋升前二次脱敏 + 人工确认）；agents 记录不含凭据/机器指纹（24 §7）。
- **访问控制**：单机文件系统信任域；**无认证/RBAC（明示）**——身份为约定级 label；控制 = 原语校验（owner/角色/不变量）+ 事后审计（24 §1 三层防御）。
- **审计**：全部状态变更进 events（NFR-005）；保留期 = run 生命周期 + export 归档。
- **合规约束**：无外部合规目标（本地开发工具）；代码与数据不出本机（无遥测；未来若加须显式 opt-in，触发时补 ASM）。
- **威胁建模**：STRIDE 表在 P3 完成（24 §1.4）。
