# 31. 用户旅途与验收手册

> 日期:2026-07-23(v0.2.3 起适用)
> **读者:用户。** 你不需要懂 sigmarun 的任何命令、文件或状态机。你的全部操作 = 在 Claude Code / Codex 的**聊天窗**里说话。
> 想看机器内部怎么转的,才去 [29](29-architecture-journey-and-quickstart.md)/[30](30-three-layer-test-playbook.md)(那是开发者/维护者视角)。

---

## 0. 你只需要知道三句话

| 你想干嘛 | 你在聊天窗里打 |
|---|---|
| 提一个需求,让 AI 拆好 | `/team-plan 给结算页加优惠券功能` |
| 开工(领一块活干掉) | `/team-do` |
| 现在啥进度?该我干嘛? | `/team-status` |

其余一切——初始化、拆任务、领任务、防撞车、写证据、异常恢复——**都是 AI 和 gateway 背后的事,你不用管**。

**高级用法(想指哪打哪时才需要):**

| 你想 | 你打 |
|---|---|
| 多个需求并行时,指定做哪个 | `/team-do RUN-0002` |
| 指定做某一块任务(定向) | `/team-do TASK-0003`(任务号跨需求重复时 AI 会先和你确认是哪个 run) |
| 固定这个窗口的身份(隔天续做同一身份) | `/team-do RUN-0002 --as 我的窗口名` |
| 做完一块别停,继续领 | 在指令后加 `--loop` |
| 让这个窗口专职当评审 / 验证员(full run,须换窗口) | `/team-review RUN-0002` / `/team-verify RUN-0002` |
| 全参数进阶入口(指定任务+角色+连跑) | `/team-dispatch RUN-0002 --task=TASK-0003 --role=reviewer --loop` |

不指定时的默认行为:恰好一个活跃需求 → 直接用;多个 → AI 列清单**问你**,绝不猜。

## 1. 一次性准备(2 分钟,每个 repo 一次)

```bash
npm i -g sigmarun
cd 你的项目
sigmarun adapter install --tool=all
```

装完聊天窗里就有 `/team-*` 命令了。**没有 init 步骤**——AI 第一次干活时发现没初始化会自己补,你永远不需要知道 init 存在。(嫌上面两条命令都多?直接对 AI 说"帮我在这个 repo 装上 sigmarun",它会替你跑。)

## 2. 日常旅途(9 个场景 = 9 条验收用例)

每个场景四栏:**你做什么 → 你会看到什么 → 你要决定什么 → ✅ 怎么算对**。判据全部是**你在聊天窗里肉眼可见**的东西——看不到就算不通过,不需要你去翻任何文件。

### S1 提需求
- **你**:`/team-plan 给结算页加优惠券输入框和校验`
- **看到**:AI 读完仓库,给你一份**编号拆分**:每块的标题、要改哪些文件、先后顺序;然后**停下来等你**。
- **决定**:"可以" / "第 2 块和第 3 块合并" / "别动数据库"。
- **✅**:AI **先给拆分、等你点头,才建需求**(不许先斩后奏);确认后报一个需求编号(RUN-xxxx)。

### S2 开工
- **你**:`/team-do`
- **看到**:AI 说"我想先做第 1 块,因为它不阻塞别人",等你点头;然后开始干活,干完汇报做了什么、改了哪些文件。
- **决定**:点头,或指定"先做第 3 块"。
- **✅**:AI 只做**被批准的那一块**,做完**主动停下汇报**,不会自作主张连着干下一块。

### S3 看进度
- **你**:`/team-status`
- **看到**:一句话状态("3 块做完 2 块,1 块等你答复")+ **"需要你处理"清单**,每条都带着"你该干什么"。
- **✅**:任何等待状态都**配着明确的下一步**,绝不出现"卡着但不知道为什么"。

### S4 开第二个窗口并行
- **你**:再开一个 Claude Code(或 Codex)窗口,同一个 repo,`/team-do`
- **看到**:第二个窗口领到的是**另一块**活;如果剩下的活跟第一个窗口改同一批文件,它会说"这块先缓,等 XX 完成"。
- **✅**:两个窗口**从不同时改同一批文件**;第二个窗口不会把第一个窗口的活抢走或改坏。

### S5 AI 问你问题
- **你**:(什么都不用做,AI 干活中遇到不确定的事)
- **看到**:AI 停下来问你("优惠券大小写敏感吗?"),或者你在 `/team-status` 里看到"需要你:回答 XX"。
- **决定**:答一句。
- **✅**:你答复后 AI **接着干**,不用重来;你不答,它**不会瞎猜着继续**。

### S6 中途改需求
- **你**:直接说"把任务二砍了" / "再加一块:错误提示要国际化" / "整个推翻重来"。
- **看到**:砍 → AI 告诉你**谁依赖任务二、砍了影响什么**,等你确认;加 → AI 给出新块的拆分待确认;推翻 → AI 先给**预览**(会取消哪些、谁正在干),你确认才执行。
- **✅**:任何**删除性动作都先预览、后执行**,绝无静默删除。

### S7 隔天回来 / 换窗口
- **你**:第二天随便开个新窗口,`/team-status`
- **看到**:需求清单、进度、半成品都还在(状态存在 repo 里,不在聊天记录里)。
- **✅**:**关窗口不丢任何东西**;新窗口无需任何"恢复"操作即可继续 `/team-do`。

### S8 收尾
- **你**:最后一块做完后照 AI 提示走(或 `/team-status`)。
- **看到**:AI 收整个需求,**明确交还**:"所有改动在你的工作区/分支上,去 commit / 开 PR;下一个需求随时 `/team-plan`"。
- **✅**:终点一定**落回 git**(改动可见、可 commit),绝不是"完成了但东西不知道在哪"。

### S9 窗口死了(断电/关机/AI 崩了)
- **你**:重开窗口 `/team-do`
- **看到**:AI 发现有个半成品("上个窗口做到一半留下的"),**停下来问你**:[接着它的改动继续] / [丢弃重来]。
- **✅**:掉线**不丢活**;接管**必须先问你**,不许静默续写别人的半成品。

## 3. 验收总表(测试时打勾用)

| # | 一句话判据(聊天窗可见) | 过/不过 |
|---|---|---|
| S1 | 建需求前必有拆分确认 | |
| S2 | 只做被批准的一块,做完停下汇报 | |
| S3 | 每个等待态都带下一步 | |
| S4 | 两窗口不同时改同一批文件 | |
| S5 | 问题答复后继续,不答不瞎干 | |
| S6 | 删除性动作先预览后执行 | |
| S7 | 关窗口不丢状态 | |
| S8 | 终点交还 git | |
| S9 | 半成品接管先问人 | |
| S0 | 全程你没被要求跑过任何 `sigmarun` 命令(装的两条除外) | |

**S0 是总纲**:测试中任何时刻,只要 AI 让你"去终端跑 `sigmarun xxx`"来完成**正常流程**(排障除外),就算产品不合格,直接记下来。

## 4. 什么时候才需要碰 CLI

只有两种情况:①**排障**——AI 报了它解决不了的环境问题(比如磁盘权限),会给你一条命令照抄;②**好奇**——想看机器内部(`sigmarun status` / `events`),那请移步 [29](29-architecture-journey-and-quickstart.md)/[30](30-three-layer-test-playbook.md) 开发者手册。
